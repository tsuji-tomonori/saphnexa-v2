# Architecture Evaluation & Review Report Skill（文献準拠：ATAM / SAAM / QAW）

## Quick start

- Use **QAW** when quality requirements are still vague and need to be turned into concrete scenarios.
- Use **ATAM** when an architecture already exists and the goal is to surface risks, non-risks, sensitivity points, and tradeoff points.
- Use **SAAM** when comparing architecture options or evaluating changeability and migration impact.
- Use **ATAM-lite** when time or evidence is limited. Be explicit about assumptions and confidence.

Load bundled files only when needed:

- `templates/input_brief.md`: ask for missing input in a structured way before or during review.
- `templates/quality_attribute_scenario.md`: draft or refine quality attribute scenarios.
- `templates/evaluation_matrix.md`: compare alternatives, scenarios, or mitigation options.
- `templates/review_report.md`: shape the final deliverable into a decision-oriented report.
- `references/literature_map.md`: use when the user asks for methodological background, literature grounding, or stronger justification of the review approach.

## 目的

このSkillは、ソフトウェア／システムアーキテクチャを、品質特性・シナリオ・設計判断・トレードオフ・リスクの観点でレビューし、意思決定に使える評価レポートを作成するためのものである。

中心となる方法は **ATAM（Architecture Tradeoff Analysis Method）** である。要求や品質特性が未整理な場合は **QAW（Quality Attribute Workshop）** の手順で品質特性シナリオを抽出し、変更容易性や候補アーキテクチャ比較が重要な場合は **SAAM（Software Architecture Analysis Method）** のシナリオ評価を併用する。投資判断やロードマップ化が必要な場合は、補助的に **CBAM（Cost Benefit Analysis Method）** 的な費用対効果整理を行う。

このSkillの成果物は「合否判定」ではなく、次の意思決定を支援するレビュー結果である。

- 品質要求が明確か
- アーキテクチャ上の設計判断が品質要求に対応しているか
- どの設計判断がリスク、感度点、トレードオフ点になっているか
- どのシナリオでアーキテクチャが弱くなるか
- どの追加検証、設計変更、プロトタイピング、運用設計が必要か

## 使用すべき場面

以下の依頼ではこのSkillを使う。

- アーキテクチャレビューをしてほしい
- ATAM、SAAM、QAWに沿って評価してほしい
- 品質特性や非機能要求に基づいて設計を評価してほしい
- 性能、可用性、セキュリティ、保守性、拡張性、運用性などのトレードオフを整理してほしい
- 技術選定、クラウド構成、マイクロサービス構成、データ基盤、認証基盤、連携方式などの設計判断をレビューしてほしい
- ステークホルダー向けのアーキテクチャ評価レポートを作ってほしい
- 複数のアーキテクチャ案を比較したい
- 既存システムの変更容易性、移行容易性、運用リスクを評価したい

## 文献ベースの前提

このSkillは、次の研究・技術文献に基づく。

- ATAM: 品質特性要求に照らしてアーキテクチャ上の設計判断の帰結を評価し、リスク、非リスク、感度点、トレードオフ点を抽出する方法。
- SAAM: シナリオを用いてアーキテクチャを評価し、直接対応できるシナリオと、設計変更を必要とする間接シナリオを区別する方法。
- QAW: アーキテクチャ完成前にステークホルダーから品質特性要求を抽出し、優先順位付きの精緻化シナリオを作る方法。
- CBAM: ATAMで得た設計判断・リスク・代替案に、コスト、利益、優先度を加えて意思決定ロードマップ化する補助的な方法。

文献上の重要な考え方を、レビュー時の規律として使う。

1. **品質特性は抽象名では評価しない。** 「高性能」「安全」「拡張しやすい」だけではなく、刺激、環境、対象、応答、応答測定を含むシナリオに落とす。
2. **アーキテクチャ評価は早期リスク発見である。** 不完全な情報から精密な数値予測をするのではなく、どこを深掘りすべきかを明確にする。
3. **トレードオフを隠さない。** ある品質を高める設計判断が、別の品質を悪化させる可能性を明示する。
4. **ステークホルダーの視点を扱う。** 開発者、運用者、利用者、事業責任者、セキュリティ担当、保守担当、外部連携先などで重要シナリオが異なるため、可能な限り視点を分ける。
5. **設計判断と品質要求を対応付ける。** 「この構成だからよい」ではなく、「この品質シナリオに対し、この設計判断がこのように効く」と説明する。
6. **リスク、非リスク、未確認事項を分ける。** 不明点をリスクと断定しない。対策済みの懸念は非リスクとして記録する。

## 入力情報

可能な範囲で、次の情報を受け取る。すべてが揃っていなくてもレビューを開始できるが、不足情報は「前提」「未確認事項」「追加質問」に分離する。

### 1. ビジネス／ミッション情報

- システム名、目的、解決したい課題
- 主要な事業目標、成功指標、制約
- 想定ユーザー、利用規模、ピーク条件
- リリース時期、予算、チーム体制
- 法規制、契約、監査、セキュリティ要件
- 既存システム、移行条件、外部連携

### 2. アーキテクチャ情報

- 構成図、コンポーネント図、データフロー図、デプロイ図
- 主要コンポーネントと責務
- コンポーネント間の依存関係、同期／非同期連携
- 採用技術、クラウド／オンプレミス、マネージドサービス
- データストア、整合性、バックアップ、リカバリ
- 認証、認可、暗号化、鍵管理、監査ログ
- 監視、ログ、アラート、SLO/SLA、運用体制
- CI/CD、リリース、ロールバック、IaC

### 3. 品質要求

- 性能：応答時間、スループット、同時接続数、処理件数、レイテンシ
- 可用性：SLA、RTO、RPO、冗長化、障害時動作
- 拡張性：ユーザー数、データ量、機能追加、マルチテナント
- セキュリティ：認証、認可、暗号化、監査、脅威、秘密情報管理
- 保守性：変更容易性、凝集度、結合度、テスト容易性、モジュール境界
- 運用性：監視、障害解析、デプロイ、ロールバック、手順化
- 相互運用性：プロトコル、スキーマ、API互換、外部連携
- コスト：初期費用、運用費、スケール時費用、ライセンス
- データ品質：整合性、鮮度、保持、削除、監査証跡

## 方法選択

依頼内容と入力情報に応じて、次のように方法を選ぶ。

| 状況 | 主に使う方法 | 目的 |
|---|---|---|
| 品質要求が曖昧、アーキテクチャが未完成 | QAW | 品質特性シナリオを抽出・精緻化する |
| アーキテクチャ案があり、品質要求に対する妥当性を見たい | ATAM | リスク、感度点、トレードオフ点を抽出する |
| 複数案を比較したい、または変更容易性を重視したい | SAAM + ATAM | 直接／間接シナリオ、変更影響、シナリオ相互作用を評価する |
| リスク対策や代替案をロードマップ化したい | ATAM + CBAM的整理 | 価値、コスト、優先度、実施順を整理する |
| 時間や情報が限られている | ATAM-lite | 重要シナリオに絞って暫定評価する |

## レビュー原則

### 1. シナリオ駆動で評価する

品質特性名だけで評価しない。必ず、次の要素を使ってシナリオ化する。

| 項目 | 説明 |
|---|---|
| シナリオID | 例：PERF-001、AVAIL-001、SEC-001 |
| ビジネス目標 | このシナリオが守る事業上・ミッション上の目的 |
| 品質特性 | 性能、可用性、セキュリティ、保守性など |
| 刺激源 | ユーザー、外部システム、攻撃者、障害、開発者、運用者など |
| 刺激 | 発生するイベントや変更要求 |
| 環境 | 通常時、ピーク時、障害時、メンテナンス時、リリース時など |
| 対象 | 影響を受けるシステム、コンポーネント、データ、インターフェース |
| 応答 | システムがどう振る舞うべきか |
| 応答測定 | 数値基準、SLA、RTO、RPO、検知時間、変更工数など |
| 質問 | 評価に必要な確認事項 |
| 論点 | 懸念、未確認事項、代替案 |

### 1.5. アーキテクチャ図は Mermaid で表現する

アーキテクチャの概要、構成要素、依存関係、データフロー、デプロイ、状態遷移、シーケンスを文書化する場合は、原則として Markdown 内の Mermaid 図を使う。

- コンテキストやコンポーネント関係は `flowchart` を使う。
- リクエスト、イベント、公開切替などの時系列は `sequenceDiagram` を使う。
- 状態遷移を評価する場合は `stateDiagram-v2` を使う。
- 図は評価対象の説明とリスク分析に必要な範囲に留め、根拠がない構成要素は「前提」または「未確認事項」として扱う。

### 2. 設計判断を品質特性に結び付ける

レビューでは、次を対応付ける。

```text
ビジネス目標
  → 品質特性
    → 品質特性シナリオ
      → アーキテクチャ上の設計判断
        → 期待される効果
        → 副作用・制約
        → リスク / 非リスク / 感度点 / トレードオフ点
```

### 3. 評価結果は証拠ベースにする

- ユーザーが提示した図、説明、コード、設計資料、運用条件を根拠として扱う。
- 根拠がない場合は、推測を「前提」と明示する。
- 数値データがない場合、レイテンシ、スループット、可用性、コストなどを断定しない。
- 「一般的に望ましい構成」と「このシステムで必要な構成」を区別する。
- ある設計がベストプラクティスから外れていても、品質シナリオに照らして問題化しないなら即リスクとはしない。

### 4. リスク分類を厳密に行う

| 分類 | 意味 |
|---|---|
| リスク | 品質シナリオの達成を阻害する可能性がある設計判断、欠落、依存、制約 |
| 非リスク | 懸念はあるが、現行設計・運用・制約下では十分に対処されている事項 |
| 感度点 | 特定の品質特性に対して影響が大きい設計パラメータや構成要素 |
| トレードオフ点 | 複数の品質特性に同時に影響し、一方を改善すると他方が悪化し得る箇所 |
| 未確認事項 | 情報不足で判断できない事項 |
| 改善機会 | 直ちに重大リスクではないが、設計・運用・開発効率を高める余地 |

## 標準レビュープロセス

### Step 0. 評価範囲と方法を決める

次を明確にする。

- 対象システム
- レビュー対象範囲
- 対象外範囲
- 評価目的
- 選択した方法：QAW / ATAM / SAAM / ATAM-lite / CBAM的整理
- 入力資料
- 既知の制約
- 不足情報

情報が不足している場合は、レビューを止めずに「暫定評価」として進める。ただし、結果の信頼度を下げ、追加で必要な情報を明示する。

### Step 1. ビジネスドライバを整理する

以下を抽出する。

- 事業上の成功条件
- 失敗した場合の影響
- 品質特性を駆動する要因
- ステークホルダーごとの関心事
- 制約：納期、予算、規制、既存システム、チームスキル、運用体制

出力例：

| ドライバID | ビジネス／ミッションドライバ | 関連品質特性 | 影響 |
|---|---|---|---|
| BD-001 | ピーク時も注文処理を止めない | 性能、可用性 | 売上・顧客体験に直結 |
| BD-002 | 個人情報を安全に扱う | セキュリティ、監査性 | 法令・信用に直結 |

### Step 2. アーキテクチャ概要を把握する

以下を簡潔に整理する。

- 主要コンポーネント
- 主要なコネクタ：HTTP、メッセージング、DB接続、イベント、ファイル連携など
- データフロー
- 配置／デプロイ構成
- 外部システム連携
- 主要なアーキテクチャスタイル・パターン
- 重要な設計判断
- 明示されていないが評価に影響する前提

必要に応じて次の視点で整理する。

| ビュー | 確認する内容 |
|---|---|
| コンテキスト | 外部利用者、外部システム、境界 |
| 機能／モジュール | 責務分割、依存関係、変更影響 |
| ランタイム | 通信、同期／非同期、障害伝播、負荷分散 |
| データ | データ所有、整合性、保持、バックアップ |
| 配置 | ノード、ネットワーク、冗長化、リージョン |
| セキュリティ | 信頼境界、認証、認可、暗号化、監査 |
| 運用 | 監視、ログ、アラート、デプロイ、復旧 |

### Step 3. アーキテクチャアプローチを特定する

設計上の主要なアプローチを列挙する。ここではまだ良否判定に飛ばず、「どの品質特性を狙った設計か」を確認する。

例：

- マイクロサービス化
- モジュラーモノリス
- イベント駆動アーキテクチャ
- API Gateway
- CQRS / Event Sourcing
- キャッシュ
- キュー／非同期処理
- レプリケーション／フェイルオーバー
- Blue-Green / Canaryリリース
- IaC
- OAuth/OIDC、RBAC、ABAC
- ゼロトラスト境界、WAF、レート制限
- データ分割、シャーディング

出力例：

| アプローチID | 設計判断／アプローチ | 狙う品質特性 | 関連シナリオ | 想定副作用 |
|---|---|---|---|---|
| AP-001 | キューによる非同期処理 | 可用性、スケーラビリティ | PERF-002, AVAIL-003 | 遅延、順序性、再処理設計が必要 |

### Step 4. Utility Treeを作成する

品質特性を階層化し、具体シナリオまで落とす。各シナリオに **重要度** と **実現難易度／リスク度** を付ける。

```text
品質特性
└── サブ特性
    └── 具体シナリオ
        ├── 重要度: 高 / 中 / 低
        └── 難易度・リスク度: 高 / 中 / 低
```

出力例：

| 品質特性 | サブ特性 | シナリオ | 重要度 | 難易度／リスク度 |
|---|---|---|---|---|
| 可用性 | 障害復旧 | DB障害時にRTO 15分以内で注文受付を復旧する | 高 | 高 |
| 性能 | ピーク処理 | セール開始後10分間、通常時の10倍のリクエストを処理する | 高 | 中 |

### Step 5. QAW的にシナリオを抽出・精緻化する

品質要求が曖昧な場合は、QAW的に次を行う。

1. ステークホルダー視点を列挙する。
2. 生シナリオをブレインストーミングする。
3. 類似シナリオを統合する。
4. 重要度で優先順位を付ける。
5. 上位シナリオを、刺激・環境・応答測定まで精緻化する。

ステークホルダー投票ができる場合は、各参加者にシナリオ数の約30%を投票枠として与える。できない場合は、事業影響、発生可能性、技術的不確実性、規制影響を用いて暫定優先度を付ける。

### Step 6. ATAM的にアーキテクチャアプローチを分析する

優先度の高いシナリオから順に、次を行う。

1. シナリオを実現するコンポーネント、コネクタ、データ、運用手順にマッピングする。
2. 関連するアーキテクチャアプローチを特定する。
3. 品質特性別の質問を投げる。
4. 回答・資料・設計判断を根拠として評価する。
5. リスク、非リスク、感度点、トレードオフ点、未確認事項を記録する。
6. 必要なら追加分析、プロトタイプ、負荷試験、脅威分析、障害訓練を提案する。

出力例：

| シナリオID | 関連アプローチ | 評価 | 根拠 | リスク／非リスク／感度点／トレードオフ点 |
|---|---|---|---|---|
| PERF-001 | キャッシュ、Read Replica | 部分適合 | キャッシュ方針はあるがTTLと無効化条件が未定 | T-001: 性能と整合性のトレードオフ |

### Step 7. SAAM的に変更シナリオを評価する

変更容易性、移行、拡張、複数案比較が重要な場合は、SAAM的に次を評価する。

| 項目 | 評価内容 |
|---|---|
| 直接シナリオ | 現行アーキテクチャで構造変更なしに対応できる |
| 間接シナリオ | コンポーネント追加・変更、コネクタ変更、データモデル変更などが必要 |
| 変更影響 | 変更されるコンポーネント、接続、データ、運用手順 |
| 変更コスト | 小 / 中 / 大、または見積もり根拠付き工数 |
| シナリオ相互作用 | 複数シナリオが同じコンポーネントに集中していないか |
| 分離関心 | 変更が局所化されているか、横断的に波及するか |

出力例：

| 変更シナリオ | 直接/間接 | 影響範囲 | 変更コスト | 相互作用 | 評価 |
|---|---|---|---|---|---|
| 決済プロバイダ追加 | 間接 | Payment Service, 認証設定, Webhook処理 | 中 | 外部連携処理に集中 | インターフェース抽象化が必要 |

### Step 8. 感度点を特定する

感度点は、ある品質特性の達成に強く影響する設計パラメータである。

例：

- DB接続プールサイズ、キュー長、スレッド数
- キャッシュTTL、キャッシュ無効化方式
- サービス分割粒度、API境界
- リトライ回数、タイムアウト、サーキットブレーカー閾値
- レプリケーション方式、フェイルオーバー時間
- 暗号化方式、鍵ローテーション周期
- 監査ログの粒度、保持期間
- データ所有境界、トランザクション境界

出力例：

| 感度点ID | 設計パラメータ | 影響する品質特性 | 影響内容 | 確認・検証方法 |
|---|---|---|---|---|
| S-001 | キャッシュTTL | 性能、整合性 | 長いTTLは性能を高めるが古いデータを返す可能性 | 仕様確認、整合性テスト |

### Step 9. トレードオフ点を特定する

トレードオフ点は、複数の品質特性に対して同時に影響する設計判断である。

代表例：

| 設計判断 | 改善し得る品質 | 悪化し得る品質 | 確認ポイント |
|---|---|---|---|
| キャッシュ | 性能、可用性 | 整合性、運用性 | TTL、無効化、監視 |
| マイクロサービス | 変更容易性、独立デプロイ | 運用性、性能、複雑性 | 境界、通信、監視、障害伝播 |
| 強い暗号化 | セキュリティ | 性能、運用性 | 鍵管理、CPU負荷、ローテーション |
| 同期API連携 | 単純性、一貫性 | 可用性、スケーラビリティ | タイムアウト、リトライ、フォールバック |
| 高冗長構成 | 可用性 | コスト、複雑性 | SLA、RTO、運用体制 |
| イベント駆動 | 疎結合、スケーラビリティ | デバッグ容易性、整合性 | 順序性、重複、再処理、観測性 |

### Step 10. リスクテーマを統合する

個別リスクを、ビジネスドライバを脅かすテーマにまとめる。

例：

| リスクテーマ | 関連リスク | 脅かされるビジネス目標 | 推奨対応 |
|---|---|---|---|
| 障害時復旧能力が設計・運用の両面で未検証 | R-001, R-004, R-007 | サービス停止時間最小化 | フェイルオーバー設計、DR訓練、RTO/RPO検証 |
| 外部連携の変更が複数箇所に波及する | R-002, R-005 | パートナー追加速度 | 連携アダプタ層、契約テスト |

### Step 11. CBAM的に対策を優先順位付けする

対策をロードマップにする場合は、次を使う。

| 対策ID | 対策 | 対応リスク | 期待効果 | コスト | 実施難易度 | 優先度 | 実施時期 |
|---|---|---|---|---|---|---|---|
| M-001 | DBフェイルオーバー手順の自動化 | R-001 | RTO短縮 | 中 | 中 | 高 | 短期 |
| M-002 | 外部連携アダプタ層の導入 | R-002 | 変更影響の局所化 | 中 | 中 | 中 | 中期 |

優先度は、次の観点で決める。

- 脅かされるビジネス目標の重要度
- リスクの重大度と発生可能性
- 対策の費用、工数、導入リスク
- 早期に学習できるか
- 他の対策の前提になるか

## 品質特性別の質問例

### 性能

- ピーク時のリクエスト量、処理件数、データ量は定義されているか。
- レイテンシ目標は平均値、p95、p99のどれか。
- 同期処理と非同期処理の境界は明確か。
- ボトルネック候補はどこか。DB、外部API、キュー、ネットワーク、CPU、メモリか。
- キャッシュやレプリカの整合性条件は定義されているか。
- 負荷試験の条件は品質シナリオと一致しているか。

### 可用性・レジリエンス

- 単一障害点はどこか。
- RTO/RPO/SLA/SLOは明確か。
- 障害検知、切り替え、復旧の責任分界は明確か。
- 外部サービス障害時にフォールバックできるか。
- リトライ、タイムアウト、サーキットブレーカーは設計されているか。
- バックアップからの復旧は検証済みか。

### セキュリティ

- 保護対象資産、脅威、信頼境界は明確か。
- 認証と認可は分離され、最小権限になっているか。
- 通信経路と保存データは暗号化されているか。
- 秘密情報、鍵、トークンの管理とローテーションは定義されているか。
- 監査ログは改ざん耐性、追跡性、保持期間を満たすか。
- 脆弱性対応、パッチ適用、依存ライブラリ管理は運用設計に含まれるか。

### 保守性・変更容易性

- コンポーネントの責務は明確か。
- 変更シナリオに対して変更が局所化されるか。
- ドメイン境界、API境界、データ所有境界は一貫しているか。
- テスト容易性、契約テスト、統合テスト、E2Eテストの役割は明確か。
- 共通処理や横断的関心事が過度に散らばっていないか。
- 技術選定がチームのスキルや運用体制に合っているか。

### 拡張性・スケーラビリティ

- 水平スケール可能な部分と難しい部分はどこか。
- 状態管理はスケールを妨げていないか。
- データ量増加時の分割、アーカイブ、インデックス戦略はあるか。
- テナント、地域、機能追加に対する拡張ポイントは明確か。
- スケール時にコストが急増する構成ではないか。

### 相互運用性

- 外部連携のプロトコル、スキーマ、バージョニングは定義されているか。
- 外部API障害、仕様変更、遅延への耐性はあるか。
- 契約テストや互換性テストは計画されているか。
- データ変換、マッピング、エラー処理の責務は局所化されているか。

### 運用性・観測性

- 監視対象、メトリクス、ログ、トレース、アラートは品質シナリオに対応しているか。
- 障害時に原因切り分けできるか。
- デプロイ、ロールバック、マイグレーション手順は明確か。
- 手作業が多い箇所はないか。
- 運用者の権限、手順、監査は定義されているか。

### コスト

- 平常時、ピーク時、障害時、成長時のコスト構造は把握されているか。
- 冗長化、ログ保持、データ転送、マネージドサービス、ライセンスの費用は妥当か。
- コスト削減が性能、可用性、セキュリティを損なっていないか。
- スケール時の限界費用が事業モデルに合っているか。

## 評価尺度

### 適合度

| 評価 | 意味 |
|---|---|
| 適合 | シナリオ達成に必要な設計判断と根拠が十分に示されている |
| 部分適合 | 大枠は妥当だが、一部の条件、測定、運用、例外処理が未定 |
| 不適合 | 現行設計ではシナリオ達成が難しい、または明確に矛盾している |
| 未確認 | 判断に必要な情報が不足している |

### 重要度

| 重要度 | 基準 |
|---|---|
| 高 | 事業継続、顧客体験、法令、重大障害、セキュリティ事故に直結する |
| 中 | 開発効率、運用効率、将来拡張、コストに大きく影響する |
| 低 | 現時点の影響が限定的、または代替手段がある |

### リスク重大度

| 重大度 | 基準 |
|---|---|
| Critical | 事業停止、重大な情報漏えい、法令違反、復旧不能に近い影響 |
| High | 主要機能停止、SLA違反、大規模な手戻り、高額コスト増 |
| Medium | 一部機能の品質低下、運用負荷増、局所的な変更困難 |
| Low | 改善余地はあるが、直近の事業影響は小さい |

### 信頼度

| 信頼度 | 基準 |
|---|---|
| 高 | 設計資料、数値条件、運用条件、検証結果が揃っている |
| 中 | 主要情報はあるが、一部は前提や推測に基づく |
| 低 | 情報不足が大きく、暫定評価である |

## レビュー実行時の応答方針

1. まず、対象システム、評価目的、入力情報の有無を確認する。
2. 入力情報が十分なら、すぐにレビューを実施する。
3. 入力情報が不足していても、ユーザーが明示的にレビューを求めている場合は、暫定レビューとして進め、前提と未確認事項を明記する。
4. 質問が必要な場合は、最小限にする。原則として、次の3点だけを優先して確認する。
   - 対象アーキテクチャの概要
   - 最重要のビジネス目標
   - 最重要の品質特性または懸念
5. 具体的なシステム資料がある場合は、一般論よりも資料に基づく評価を優先する。
6. 判断根拠が薄い場合は、断定ではなく「可能性」「要確認」「暫定評価」と表現する。
7. 改善提案は、短期・中期・長期に分け、実行可能性を重視する。
8. 必要に応じて、LLMはシナリオ案、質問案、リスク候補の生成を支援するが、最終評価は入力資料・根拠・ステークホルダー確認に基づくものとして扱う。
