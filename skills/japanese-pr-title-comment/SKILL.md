# Japanese PR Title and Comment Skill

## 目的

Pull Request のタイトル、本文、レビューコメント、通常コメントを日本語で読みやすく作成する。
変更内容、背景、確認内容、影響範囲を明確にし、レビュー担当者が判断しやすい PR にする。

## 起動条件

ユーザーが次のような依頼をしたら、このスキルを使う。

- PR タイトルを作る、直す、提案する
- PR 本文、PR コメント、レビューコメント、説明文を作る
- GitHub の Pull Request を作成、更新、説明する
- `gh pr create`、GitHub app、PR テンプレート、PR description に関する作業をする
- `git diff`、変更ファイル一覧、コミット内容、作業レポートから PR 文面を作る

## 基本ルール

1. **日本語を基本にする**
   - PR タイトル、本文、コメント、レビューコメントは日本語で書く。
   - ブランチ名、ファイルパス、コマンド、API 名、型名、関数名、issue 番号は原文を維持する。

2. **タイトルは具体的にする**
   - 何をどう変えたかが分かる短い日本語にする。
   - 「修正」「対応」「更新」だけで終わらせない。
   - 目安は 30〜60 文字程度にする。

3. **本文は PR テンプレートに沿う**
   - `.github/pull_request_template.md` の見出しを優先する。
   - 空欄を残す場合は、未確認理由や該当なしであることを明記する。
   - 確認していないテストにチェックを入れない。

4. **レビューコメントは行動可能にする**
   - 指摘は、問題、理由、提案を分けて書く。
   - 推測だけで断定しない。
   - 必要に応じてファイルパス、行番号、再現条件を含める。

5. **作業レポートがある場合は活用する**
   - `reports/working/*.md`、`reports/bugs/*.md`、または同等のレポートがある場合は内容を確認する。
   - PR 本文には、実施作業、成果物、未対応事項、制約、リスクを要約して反映する。

## PR 本文の書き方

```markdown
# 概要

## 変更内容

- 変更した主なファイルや機能を箇条書きで整理する

## 背景・目的

- なぜこの変更が必要かを簡潔に書く

## 確認内容

- [x] 実行したテストや確認
- [ ] 未実施の確認

## 影響範囲

- 利用者、API、画面、運用、CI などへの影響を書く

## 補足

- レビュー時に見てほしい点、未対応、既知の制約を書く
```

## 避ける表現

- 「いい感じにしました」
- 「いろいろ修正」
- 「確認お願いします」だけの本文
- 実施していないテストを実施済みとして書く
- 英語タイトルや英語だけの PR コメント
