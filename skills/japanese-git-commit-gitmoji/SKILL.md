# Japanese Git Commit Message with gitmoji Skill

## 目的

Git の変更内容、`git diff`、ステージ済みファイル、PR 内容、またはユーザーの短い説明から、日本語で読みやすく、慣例的で、自動化にも向いたコミットメッセージを作成する。

このスキルでは、以下を組み合わせる。

- Conventional Commits の `type(scope): description` 形式
- gitmoji による変更意図の視覚的表現
- 日本語の簡潔な要約
- 既にステージ済みの作業レポートがある場合の本文サマリ
- 必要に応じた footer・BREAKING CHANGE 表記

## 起動条件

ユーザーが次のような依頼をしたら、このスキルを使う。

- 「コミットコメントを作って」
- 「commit message を作成して」
- 「git commit 用のコメントを日本語で」
- 「この diff からコミットメッセージを考えて」
- 「gitmoji 付きでコミットメッセージを書いて」
- `git diff`、`git status`、変更ファイル一覧、PR 内容を提示した場合

Git の正式な用語としては「commit comment」ではなく「commit message」を使う。ただし、ユーザーが「コメント」と言った場合も commit message として扱う。

## 基本出力形式

原則として、1 行目は次の形式にする。

```text
<emoji> <type>(<scope>): <日本語の要約>
```

scope が不要または不明な場合は省略する。

```text
<emoji> <type>: <日本語の要約>
```

破壊的変更がある場合は `!` または `BREAKING CHANGE:` を使う。

```text
💥 feat(api)!: ユーザー取得APIのレスポンス形式を変更

BREAKING CHANGE: `name` を `displayName` に変更。
```

## 出力ルール

1. **日本語を基本にする**
   - 要約・本文は日本語で書く。
   - `type`、`scope`、API 名、クラス名、関数名、issue 参照、footer は英語・識別子を維持する。

2. **1 行目は短く具体的にする**
   - 目安は 50〜72 文字以内。
   - 「修正」「変更」「対応」だけで終わらせない。
   - 何がどう変わったかを示す。

3. **敬体を避ける**
   - 避ける: `ログイン処理を修正しました`
   - 推奨: `ログイン失敗時のエラー表示を修正`

4. **変更の主目的を 1 つに絞る**
   - 機能追加とバグ修正が混在する場合は、可能なら複数コミットへの分割を提案する。
   - 分割できない場合は、最も重要な変更を type と emoji に反映する。
   - 実際に `git commit` する場合、分割できる複数目的の変更を 1 commit にまとめない。
   - 1 行目の要約で `〜と〜`、`〜および〜`、`〜・〜`、`〜/〜` によって複数の修正目的を並列している場合は、原則として commit 分割のサインとして扱う。
   - 例外は、1 つの概念名・機能名・固定表現に含まれる接続だけとする。判断に迷う場合は分割を優先する。

5. **本文は必要なときだけ付ける**
   - 1 行で十分なら本文は不要。
   - 理由・影響・移行手順・制約がある場合は本文を追加する。
   - 実際に `git commit` する場合、commit 対象に作業レポートが含まれるなら、本文は原則として付ける。
   - commit message 作成や `git commit` のためだけに、新しい作業レポートを作成しない。

6. **差分から推測しすぎない**
   - 変更内容が不明な場合は、暫定候補であることを明示する。
   - ユーザーに `git diff --staged` や変更概要の提示を促してよい。

7. **commit 対象の作業レポートを反映する**
   - 実際に `git commit` する前に、`git diff --cached --name-only` などでステージ済みファイルを確認する。
   - ステージ済みファイルに `reports/working/*.md`、`reports/bugs/*.md`、または同等の作業・障害レポートが含まれる場合は、そのレポート本文を確認する。
   - commit message の本文には、確認した作業レポートの内容をもとに、主な作業・判断・成果物・注意点を日本語の箇条書きで要約する。
   - 箇条書きは 3〜6 項目を目安にし、作業ログの羅列ではなく commit の理解に役立つ内容へ圧縮する。
   - 複数レポートが含まれる場合は、重複を統合し、今回の commit に直接関係する内容を優先する。
   - ステージ済みファイルに作業レポートが含まれない場合は、commit 作業のためだけに新規レポートを作成しない。

## 生成手順

1. 変更内容を読み取り、主目的を決める。
2. 変更が複数目的に分かれるか確認する。
   - 分かれる場合は、目的別に stage 範囲と commit message を分ける。
   - 1 行目が `A と B を追加` のようになるなら、まず `A` と `B` を別 commit にできないか検討する。
   - 作業レポートが含まれる場合も、レポートの存在だけを理由に無関係な変更をまとめない。
3. `type` を選ぶ。
4. 対応する gitmoji を選ぶ。
5. `scope` を決める。
   - 例: `auth`, `api`, `ui`, `db`, `deps`, `ci`, `docs`, `test`, `config`
6. 日本語の要約を作る。
7. 要約が複数目的を接続していないか確認する。
   - `〜と〜` で複数目的を並べている場合は、commit を分割する。
   - 分割しない場合は、なぜ 1 commit として不可分なのか本文で説明できる必要がある。
8. ステージ済みファイルに作業レポートが含まれるか確認する。
   - 含まれる場合はレポートを読み、本文用の箇条書きサマリを作る。
   - 含まれない場合でも、変更が大きい・複数目的・リスクや制約があるなら本文を検討する。
9. 可能なら 3 候補を出す。
   - 推奨案
   - 短め案
   - 詳細め案
10. 必要に応じて本文付き版を出す。
11. 破壊的変更・issue 参照・revert がある場合は footer を付ける。

## type の選び方

| type | 用途 |
|---|---|
| `feat` | 新機能、既存機能へのユーザー可視の追加 |
| `fix` | バグ修正、不具合解消、例外・失敗ケースへの対応 |
| `perf` | 速度、メモリ、クエリ数などの性能改善 |
| `refactor` | 外部挙動を変えない構造改善 |
| `style` | コード整形、フォーマット、UI/CSS/見た目の変更 |
| `docs` | README、仕様書、コメント、ドキュメント更新 |
| `test` | テスト追加、テスト修正、テスト安定化 |
| `build` | ビルド設定、バンドル、パッケージ生成 |
| `ci` | CI/CD、GitHub Actions、ワークフロー修正 |
| `chore` | 開発環境、設定、依存関係、定型作業 |
| `revert` | 既存コミットの取り消し |
| `release` | リリース、バージョンタグ、公開作業 |

## 悪い例と良い例

### 曖昧すぎる

悪い:

```text
🐛 fix: バグ修正
```

良い:

```text
🐛 fix(auth): トークン期限切れ時の再ログイン処理を修正
```

### 英語と日本語が不自然に混在

悪い:

```text
✨ feat: add ログイン機能
```

良い:

```text
✨ feat(auth): パスワードレスログインを追加
```

### scope が広すぎる

悪い:

```text
♻️ refactor(app): コードを整理
```

良い:

```text
♻️ refactor(user): プロフィール更新処理を service 層へ分離
```

### 作業ログになっている

悪い:

```text
📝 docs: READMEを書いた
```

良い:

```text
📝 docs(readme): セットアップ手順を追加
```

### 複数目的をまとめている

悪い:

```text
📝 docs: SWEBOKテンプレート文書とcommit運用を追加
```

理由: `docs` テンプレート追加と commit 運用変更は目的が異なるため、1 commit にまとめない。

良い:

```text
📝 docs: SWEBOKテンプレート文書を追加
🔧 chore(commit): 既存レポート参照ルールを追加
```

複数目的の変更を見つけた場合は、候補メッセージだけでなく分割案も示す。

## 出力テンプレート

ユーザーが変更内容を渡した場合は、次の形式で返す。

```text
推奨:
`🐛 fix(auth): トークン更新失敗時の再ログイン処理を修正`

候補:
1. `🐛 fix(auth): トークン期限切れ時の再ログイン処理を修正`
2. `🐛 fix(auth): 認証トークン更新失敗時の例外を防止`
3. `🐛 fix(auth): セッション期限切れ時のエラーハンドリングを改善`
```

本文が必要な場合:

````text
推奨:
`⚡️ perf(search): 検索結果の初期表示を高速化`

本文付き:
```commit
⚡️ perf(search): 検索結果の初期表示を高速化

- N+1 クエリを解消し、一覧取得時の DB アクセスを削減。
- レスポンス形式は既存仕様を維持。
```
````

作業レポートを含む commit の場合:

````text
推奨:
`🔧 chore: AI作業基盤とリポジトリ設定を追加`

本文付き:
```commit
🔧 chore: AI作業基盤とリポジトリ設定を追加

- リポジトリローカル skill と agent 定義を追加。
- `.codex` 関連の環境ファイルを `.gitignore` に追加。
- ステージ済み作業レポートに記録された制約と判断を commit 本文へ反映。
```
````

複数目的が混在する場合:

````text
分割推奨:

1. `📝 docs: SWEBOKテンプレート文書を追加`
```commit
📝 docs: SWEBOKテンプレート文書を追加

- docs 各葉ディレクトリに初期テンプレートを追加。
- docs テンプレート生成 skill と生成スクリプトを更新。
```

2. `🔧 chore(commit): 既存レポート参照ルールを追加`
```commit
🔧 chore(commit): 既存レポート参照ルールを追加

- commit 対象に既存レポートが含まれる場合のみ確認するルールを追加。
- commit 作業だけでは新規レポートを作成しないことを明記。
```
````

issue 参照がある場合:

```text
🐛 fix(api): null 応答時の例外を防止

Refs: #123
```

破壊的変更がある場合:

```text
💥 feat(api)!: ユーザー取得APIのレスポンス形式を変更

BREAKING CHANGE: `name` を `displayName` に変更。
```

revert の場合:

```text
⏪️ revert: OAuthログイン追加を取り消し

Refs: abc1234
```

## git commit コマンド例

1 行メッセージ:

```bash
git commit -m "✨ feat(auth): パスワードレスログインを追加"
```

本文付き:

```bash
git commit -m "⚡️ perf(search): 検索結果の初期表示を高速化" \
  -m "- N+1 クエリを解消し、一覧取得時の DB アクセスを削減。" \
  -m "- レスポンス形式は既存仕様を維持。"
```

破壊的変更:

```bash
git commit -m "💥 feat(api)!: ユーザー取得APIのレスポンス形式を変更" \
  -m "BREAKING CHANGE: name を displayName に変更。"
```

## ユーザーから情報が少ない場合

変更内容が不足している場合は、次のように返す。

```text
変更内容がまだ分からないため、暫定の形式だけ提示します。

`✨ feat(scope): 変更内容を追加`

より正確に作るには、変更概要・変更ファイル・`git diff --staged` のいずれかを貼ってください。
```

ただし、ユーザーが急いでいる場合や情報が断片的でも十分判断できる場合は、最善の候補を出す。

## 最終チェックリスト

出力前に確認する。

- [ ] 先頭に適切な gitmoji がある
- [ ] `type` が変更目的と一致している
- [ ] scope が具体的で、不要なら省略されている
- [ ] 日本語要約が具体的で短い
- [ ] 「修正」「対応」「変更」だけで終わっていない
- [ ] 1 行目が `〜と〜` などで複数目的をまとめていない
- [ ] 分割できる複数目的の変更は、目的別 commit に分けている
- [ ] commit 対象に作業レポートが含まれる場合、その内容を確認している
- [ ] 作業レポートを確認した場合、本文に日本語の箇条書きサマリがある
- [ ] 本文・footer が必要な場合だけ追加されている
- [ ] 破壊的変更なら `!` または `BREAKING CHANGE:` がある
- [ ] 複数目的の変更なら分割コミットを提案している
