# Post Task Fit Report Skill

## 目的

このスキルは、作業完了後に以下を明確に整理したレポートを作成する。

1. どういった指示があったか
2. その指示に対して何を検討し、どう判断したか
3. 結果としてどのような作業を行ったか
4. どの成果物を出したか
5. 成果物が指示にどれくらい適合しているか
6. 未対応事項、制約、リスク、改善余地は何か

このレポートは、作業の透明性、検収性、再現性を高めるためのものである。

## 使用タイミング

このスキルは次のタイミングで使用する。

- 実作業を伴うタスクが完了し、作業レポートが必要なとき。
- ユーザーが「作業完了後にレポートして」「fitを評価して」「指示との適合度を出して」「完了報告を書いて」などと依頼したとき
- 成果物の提出時に、指示との対応関係を明示する必要があるとき
- 複数の指示、制約、判断、成果物が存在し、検収しやすい形に整理する必要があるとき
- 途中で指示変更、追加要件、制約、代替判断が発生したとき

リポジトリの運用で求められる場合は、主作業を終えたあと最終回答の前にこのスキルを実行し、`reports/working/` など指定先にレポートを保存する。

## 使用しないタイミング

次の場合、このスキルだけを理由に新規レポートを作成しない。

- `git commit` の実行、commit message 作成、commit 用コメント作成だけを依頼されたとき。
- `git diff`、`git status`、ステージ済み差分確認を commit message 作成のためだけに行うとき。
- 既存の作業レポートを commit message や PR 文面の材料として読むだけのとき。

既に `reports/working/*.md` や `reports/bugs/*.md` がステージ済みの場合は、その内容を必要に応じて commit message へ反映する。ただし、commit 作業のためだけに新しい作業レポートを追加しない。

## 基本方針

- 報告は、ユーザーが検収しやすいように簡潔かつ具体的に書く。
- 指示、判断、実施内容、成果物、適合度を混同しない。
- 内部の逐語的な思考過程や隠れた推論は開示しない。代わりに、ユーザーに説明可能な「検討・判断の要約」を示す。
- できたこと、できなかったこと、確認できなかったことを正直に分けて書く。
- 成果物がファイルの場合は、ファイル名、形式、内容、用途を明示する。
- 情報源を使った場合は、必要に応じて出典や根拠を示す。
- 不確実性がある場合は、断定せず「未確認」「推定」「制約により未検証」などと明記する。
- システム上・安全上・技術上の制約で実施できなかったことは、隠さず簡潔に説明する。
- 作業レポートは原則としてチャット本文だけで完結させず、`reports/working/` 配下に Markdown ファイルとして保存する。
- リポジトリ方針またはユーザー指示がある場合は、実作業を伴うタスクごとに 1 件のレポートを残す。
- Markdown の強制改行用の末尾スペースは使わない。各項目は通常の改行または箇条書きで分け、pre-commit の `trailing-whitespace` hook で中断されない形にする。

## 出力先ルール

- レポート出力先は原則 `reports/working/` とする。
- `reports/working/` が存在しない場合は作成してから保存する。
- 保存形式は Markdown (`.md`) とする。
- ファイル名は `YYYYMMDD-HHMM-<task-summary>.md` を推奨する。
- `<task-summary>` は ASCII の小文字とハイフンを優先し、短く要点が分かる名称にする。
- 適切な task summary を付けにくい場合は `YYYYMMDD-HHMM-work-report.md` を使う。
- 最終回答では、生成したレポートの保存先パスを明示する。

## レポート作成手順

### 1. 出力先の決定

保存先ファイルを最初に決める。原則として `reports/working/` 配下に Markdown ファイルを作成し、そのパスを成果物として扱う。

例:

- `reports/working/20260430-0930-skill-install-report.md`
- `reports/working/20260430-0930-work-report.md`

### 2. 指示の抽出

ユーザーからの明示的な依頼、追加依頼、制約、希望形式、禁止事項を抽出する。

抽出対象:

- 主目的
- 成果物の種類
- 形式・トーン・分量・言語
- 期限・対象範囲・前提条件
- 使用すべき情報源、参照資料、既存ファイル
- 避けるべき内容や制約
- 途中で追加・変更された指示

### 3. 要件への分解

指示を検収可能な要件に分解する。

例:

| 要件ID | 要件 | 重要度 | 対応状況 |
|---|---|---:|---|
| R1 | 作業後レポートを出す | 高 | 対応 |
| R2 | 検討・判断を説明する | 高 | 要約で対応 |
| R3 | 指示へのfitを評価する | 高 | スコアと理由で対応 |

### 4. 検討・判断の要約

以下を、内部思考の逐語的開示ではなく、説明可能な判断根拠としてまとめる。

- どの要件を優先したか
- 曖昧な点をどう解釈したか
- 採用した方針・構成・手法
- 採用しなかった選択肢と理由
- 制約により調整した点
- 品質確認の観点

### 5. 実施作業の整理

実際に行った作業を時系列またはカテゴリ別に記載する。

例:

- 要件整理
- 情報収集・確認
- 構成設計
- 本文作成
- ファイル生成
- 表記・整合性確認
- 成果物リンクの整理

### 6. 成果物の一覧化

成果物がある場合、次を明示する。

| 成果物 | 形式 | 内容 | 指示との対応 |
|---|---|---|---|
| `reports/working/example-report.md` | Markdown | 完了レポート本文 | レポート出力要件に対応 |

成果物がファイルではなく本文回答の場合も、「この回答本文」などと明記する。
ただし、このスキルでは可能な限りファイル成果物を優先する。

### 7. Fit評価

指示への適合度を、定量評価と定性評価の両方で示す。

推奨評価軸:

| 評価軸 | 観点 |
|---|---|
| 指示網羅性 | 明示された依頼をどれだけ満たしたか |
| 制約遵守 | 形式、分量、言語、禁止事項を守ったか |
| 成果物品質 | 実用性、明確性、完成度、整合性 |
| 説明責任 | 判断理由、未対応事項、限界を説明したか |
| 検収容易性 | ユーザーが確認・修正しやすい形か |

スコア目安:

| スコア | 適合度 | 定義 |
|---:|---:|---|
| 5 | 95〜100% | 指示をほぼ完全に満たし、未対応や不確実性が軽微 |
| 4 | 80〜94% | 主要要件を満たすが、軽微な不足・未検証事項がある |
| 3 | 60〜79% | 中核は満たすが、明確な不足や改善余地がある |
| 2 | 40〜59% | 一部対応に留まり、重要要件に不足がある |
| 1 | 1〜39% | 指示との乖離が大きい |
| 0 | 0% | 指示に実質的に対応していない |

最終評価は次の形式で書く。

```text
総合fit: 4.5 / 5.0（約90%）
理由: 主要要件は満たしたが、〇〇は未検証のため満点ではない。
```

### 8. 保存と最終報告

- 完成したレポートを `reports/working/` 配下の Markdown ファイルに保存する。
- 保存後、可能であればリポジトリで定義済みのドキュメント検証、または対象ファイルへの末尾空白チェックを実行する。
- 最終回答では、保存先パス、成果物名、未検証事項の有無を簡潔に示す。

### 9. 未対応・制約・リスク

以下を分けて書く。

- 未対応事項: 指示に含まれていたが実施できなかったもの
- 制約: ツール、情報、時間、ファイル、権限などの制約
- リスク: ユーザー確認が必要な点、解釈違いの可能性、品質上の懸念
- 改善案: 次に行うとよい修正・追加作業

## 簡易版テンプレート

短い作業やチャット本文だけの成果物では、以下の簡易版を使う。

```markdown

**検討・判断:**
```

## 評価時の注意

- fit評価は自己肯定的に盛らない。根拠が弱ければ点を下げる。
- 「ユーザーが求めたもの」と「実際に提供したもの」の差分を必ず見る。
- 成果物が未検証の場合は、品質評価に反映する。
- ユーザーの明示指示と上位制約が衝突した場合は、上位制約を優先し、その影響を説明する。
- 不完全な成果物でも、どこまで対応済みかを明確にする。

## 出力例

```markdown
# 作業完了レポート

保存先: `reports/working/20260430-0930-post-task-fit-report-skill.md`

## 1. 受けた指示

- 作業完了後に、指示内容、検討・判断、実施作業、成果物のfitを報告できるスキルを作成すること。
- すぐ使える形にすること。

## 2. 要件整理

| 要件ID | 指示・要件 | 重要度 | 対応状況 |
|---|---|---:|---|
| R1 | 指示内容を整理する | 高 | 対応 |
| R2 | 検討・判断を説明する | 高 | 対応 |
| R3 | 実施作業を示す | 高 | 対応 |
| R4 | 成果物のfitを評価する | 高 | 対応 |
| R5 | スキルとして再利用可能にする | 高 | 対応 |

## 3. 検討・判断したこと

- 検収しやすいよう、指示、判断、作業、成果物、fit評価を分離した。
- 内部思考の逐語的開示ではなく、判断根拠の要約を出す設計にした。
- 短い作業にも使えるよう、標準版と簡易版のテンプレートを用意した。

## 4. 実施した作業

- スキルの目的、使用タイミング、基本方針を定義した。
- レポート作成手順を設計した。
- fit評価の評価軸とスコア目安を作成した。
- 標準テンプレートと簡易テンプレートを作成した。

## 5. 成果物

| 成果物 | 形式 | 内容 | 指示との対応 |
|---|---|---|---|
| `skills/post-task-fit-report/SKILL.md` | Markdown | 作業後fitレポート生成スキル | 依頼内容に対応 |
| `reports/working/20260430-0930-post-task-fit-report-skill.md` | Markdown | 作業完了レポート | レポート出力要件に対応 |

## 6. 指示へのfit評価

| 評価軸 | 評価 | 理由 |
|---|---:|---|
| 指示網羅性 | 5/5 | 指示内容、検討、作業、fit評価をすべて含む |
| 制約遵守 | 5/5 | 日本語で再利用可能な形式にした |
| 成果物品質 | 4.5/5 | 実運用可能だが、利用環境ごとの調整余地はある |
| 説明責任 | 5/5 | 未対応や制約を明示する設計を含む |
| 検収容易性 | 5/5 | 表形式とテンプレートで確認しやすい |

**総合fit: 4.9/5（約98%）**

理由: 依頼されたレポート生成機能をスキルとして定義し、運用テンプレートまで含めたため。利用環境固有の設定は未指定のため、一般的なMarkdown形式で作成した。

## 7. 未対応・制約・リスク

- 未対応: 特定プラットフォーム固有の登録手順は未記載。
- 制約: 実際の運用環境に応じて命名や配置は調整が必要。
- リスク: fit評価の厳しさは運用者によって差が出る可能性がある。
```
