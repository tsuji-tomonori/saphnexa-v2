# なぜなぜ分析 Skill

## 目的

発生した問題について、観測事実から因果関係をたどり、再発防止につながる改善点を特定する。分析のゴールは「誰が悪いか」ではなく、「同じ状況で普通の人が同じ失敗をしにくい仕組みに変えること」である。

## 使用する場面

このスキルは、次の条件を満たす問題に使う。

- 問題事象がすでに発生しており、発生日時、場所、対象、影響がある程度特定できる。
- 標準、計画、期待値、SLA、仕様、通常状態との差異が説明できる。
- 再発防止、品質改善、プロセス改善、障害予防が目的である。
- 迅速な初期分析が必要だが、結論には証拠と検証を必要とする。

次の条件では、なぜなぜ分析単独で結論を出さない。

- 人命、安全、医療、法規制、重大な財務影響を伴う。
- 複数部署、複数システム、外部委託先、顧客行動などが絡む。
- 事実が不足しており、仮説と確認済み事実の区別が難しい。
- 単線的な原因ではなく、複数要因の相互作用が疑われる。

この場合は、RCA、FTA、FMEA、WBA、HFACS、SEIPS、変更分析、タスク分析、ヒューマンファクター分析などへ拡張する。

## 基本原則

1. **問題文は原因を含めず、観測事実で書く。** 例：「出荷検査でA製品の寸法が規格上限を0.3mm超えた」。悪い例：「作業者が確認を怠ったため不良が出た」。
2. **なぜは回数固定ではない。** 5回は目安であり、根本原因に到達したかどうかは、証拠、因果妥当性、対策可能性で判断する。
3. **1本の鎖に押し込まない。** 直接原因が複数ある場合は、分岐した因果ツリーにする。
4. **各「なぜ」は証拠で支える。** 記録、現物、ログ、測定値、インタビュー、手順書、教育記録、変更履歴、環境条件などを根拠にする。
5. **人のミスで止めない。** 「失念」「確認不足」「判断ミス」は終点ではなく、作業条件、情報設計、教育、負荷、標準、文化、管理の問いに進む。
6. **局所要因と組織要因を分けて見る。** 局所要因は人の行動に直接影響した条件、組織要因は局所要因を適切な状態に保つマネジメントの状態である。
7. **対策は原因に対応させる。** 対策は、どの因果リンクまたは原因を断つのかを明示する。
8. **弱い対策だけで終えない。** 注意喚起、教育、周知、再徹底は補助策に留め、可能な限り仕組み、設計、標準、検出、フェイルセーフ、権限、リソース配分を変える。

## 実行手順

### 0. トリアージ

まず、分析深度を決める。

| 判定観点 | 軽量なぜなぜでよい | 拡張RCAが必要 |
|---|---|---|
| 影響 | 小さく局所的 | 人身、安全、規制、顧客重大影響 |
| 因果 | 単純・再現可能 | 複数要因、相互作用、不確実性大 |
| データ | 現物・ログ・記録あり | データ欠落、証言依存、矛盾あり |
| 対策 | 現場裁量で改善可能 | 組織横断・投資・標準変更が必要 |

重大または複雑な場合は、なぜなぜ分析を「初期仮説整理」として使い、正式なRCAへ移行する。

### 1. 問題を定義する

次の形式で書く。

> いつ、どこで、何が、期待値・標準・仕様からどれだけ外れ、どの影響を生んだか。

確認する項目：

- What：何が起きたか
- Where：どこで起きたか
- When：いつ起きたか
- Extent：どれくらい起きたか、何件か、再現性はあるか
- Standard：本来どうあるべきだったか
- Boundary：何は今回の分析対象外か

### 2. 事実を集める

最低限、次を集める。

- 現物、写真、測定値、ログ、監視データ
- 作業手順書、チェックリスト、仕様書、標準、SLA
- 変更履歴、リリース履歴、設備保全履歴、教育履歴
- 当事者・関係者インタビュー。ただし、責任追及ではなく「作業条件を理解する」目的で行う。
- タイムライン：事前条件、発生、検知、対応、復旧、再発防止まで

事実・推測・未確認を分ける。

### 3. 直接原因を分岐で出す

最初の問いは「なぜこの問題が発生したか」ではなく、次のように具体化する。

- なぜ標準から外れたか。
- なぜ異常が検出されなかったか。
- なぜ影響が拡大したか。
- なぜ復旧・是正が遅れたか。

発生原因、流出原因、検知遅れ、影響拡大を分けると、単線化を防げる。

### 4. 3レーンで掘り下げる

各原因について、3つのレーンで「なぜ」を続ける。

#### A. 発生・技術・プロセスレーン

- どの工程、条件、入力、設備、材料、データ、設計、環境が関係したか。
- 通常との差異は何か。
- 変更点は何か。
- 測定・検査・監視は十分だったか。
- 標準と実作業は一致していたか。

#### B. 人・作業条件レーン

- 当時、作業者・担当者に見えていた情報は何か。
- 手順は実行しやすかったか。曖昧さ、例外、分岐、過負荷はあったか。
- ツールやUIは誤操作を誘発していないか。
- 時間圧、割込み、騒音、疲労、交代、経験差はあったか。
- 教育は「知っている」だけでなく「その状況で実行できる」水準だったか。

#### C. システム・組織レーン

- 標準・手順・教育・監査・レビュー・変更管理はどのように維持されていたか。
- リスクが既知だったなら、なぜ対策されていなかったか。
- 必要なリソース、時間、権限、設備、情報が不足していなかったか。
- 目標、評価、コスト、納期、文化が望ましくない行動を促していなかったか。
- 類似事象が過去にあったなら、なぜ学習が組織に残らなかったか。

### 5. 因果リンクを検証する

各リンクについて、次のテストを行う。

| テスト | 合格基準 |
|---|---|
| 証拠テスト | 観測事実または信頼できる記録がある |
| 反事実テスト | その原因がなければ結果は起きなかった、または起きにくかったと説明できる |
| 十分性テスト | 挙げた原因群が揃うと、問題発生を合理的に説明できる |
| 非飛躍テスト | 「忙しかった → ミスした」のような粗い飛躍がなく、中間条件が説明されている |
| 対策可能性テスト | その原因に対して、現実的な改善策が設計できる |
| 非責任追及テスト | 個人属性ではなく、再発を防ぐ条件変更として表現されている |

不合格なら、事実収集へ戻る。

### 6. 根本原因を整理する

根本原因は「最も深い原因」ではなく、「再発リスクを大きく下げるために変えるべき原因」として整理する。

分類する。

- 直接原因：問題を直接引き起こした条件
- 流出原因：発生した問題を検知・遮断できなかった条件
- 局所要因：人の行動に直接影響した作業条件
- 組織要因：局所要因を悪い状態に放置した管理・標準・文化・資源配分
- 未確認仮説：証拠不足のため保留する原因候補

### 7. 対策を設計する

対策は、原因ごとに「除去」「低減」「検知」「影響緩和」「標準化」に分けて設計する。

強い順の目安：

1. 危険・不良・誤操作が起きない設計にする。
2. 物理的・システム的なガード、インターロック、権限制御、自動検知を入れる。
3. 作業フロー、標準、レビュー、承認、変更管理を変える。
4. チェックリスト、視覚化、アラート、ダブルチェックなど認知支援を入れる。
5. 教育、周知、注意喚起を行う。

教育・周知が必要な場合でも、単独対策にしない。必ず「なぜ教育だけで防げるのか」「教育効果をどう測るか」を書く。

各対策は以下を満たす。

- 対応する原因が明確である。
- 実施責任者、期限、完了条件がある。
- 効果指標、監視方法、レビュー日がある。
- 副作用、コスト、運用負荷、他工程への影響を検討している。

### 8. 効果を検証し、標準化する

- 再発件数、検出率、リードタイム、不良率、アラート件数、監査結果などで測る。
- 対策後に同種・類似事象が減ったかを確認する。
- 標準、手順、教育資料、設計ルール、監査項目に反映する。
- 類似プロセスへ水平展開する。
- 未確認仮説が残る場合、追加調査計画を残す。

## 終了条件

分析は、次を満たしたときに完了とする。

- 問題文が原因を含まず、事実で記述されている。
- 因果ツリーの各リンクに証拠がある。
- 複数原因がある場合、分岐として表現されている。
- 人のミスで止まらず、作業条件・管理条件まで掘れている。
- 原因と対策が1対1または多対1で対応している。
- 少なくとも1つは中程度以上の仕組み変更・検出強化・標準変更がある。
- 対策のオーナー、期限、効果指標、フォローアップ日がある。

## 禁止パターン

- 「作業者の不注意」「意識不足」「確認不足」を最終原因にする。
- 事実確認なしに会議室で原因を決める。
- 5回で止める、または5回まで無理に続ける。
- 複数原因を1本の鎖に押し込む。
- 「教育を徹底する」「注意喚起する」「ルールを守らせる」だけで終える。
- 対策の効果測定を定義しない。
- 個人懲戒と安全・品質分析を混同する。

## 標準アウトプット

```text
1. 問題文
2. 影響範囲
3. 事実・証拠一覧
4. タイムライン
5. 因果ツリー
6. 根本原因一覧
   - 直接原因
   - 流出原因
   - 局所要因
   - 組織要因
7. 対策案
   - 原因との対応
   - 強さ
   - オーナー
   - 期限
   - 効果指標
8. 残課題・未確認仮説
9. フォローアップ計画
```
